1.1. Re-Running PIPE code (generating landscapes)

1.1.1. Re-generating simProt files

A simProt file for a protein sequence represents all interactions of this protein to all other proteins. This is directly loaded in memory while calculating the landscape to significantly reduce the run time of pipe for each input-pair
  • lab/Graham: 8 CPUs
    • ~16 hrs runtime
  • AWS: 196 CPUs (with 4GB memory each CPUs)
    • ~10 minutes
  • Output: one  .db  file per protein sequence

1.1.2. Re-generating Landscapes

Once the  simProt  files are generated, we run the pipe algo to generate landscapes for each input pair
  • lab/Graham:  8 CPUs
    • ~30 minutes
  • AWS: on 196 CPUs (with 4GB memory each CPUs)
    • ~1 minute
  • Output : one landscape ( .mat  file) for each input pair

2.1. Understanding Similarity-Weighted (SW) through 3-D plots

2.1.1. Original-PIPE landscape

These are the direct results of PIPE (in the form of .mat files). We will apply various approaches in the upcoming sections on this landscapes to predict binding sites for PPI


        • One example of landscape (.mat file)

2.1.2. norm_factor

While calculating the SW landscape we normalize the original landscape value by a factor completely depending on the position/biding-site. This is not uniform for the entire image, meaning norm-factor is not a constant value for the image but for each pixel i,j its value is simProt(i)*simProt(j) 



        • one example of what is the value of norm_factor for each pixel

2.1.3. SW-PIPE landscapes

This represents landscape values right after SW normalization. In summary we can say that
SW-PIPE-landscape = Original-PIPE-landscape / norm_factor

        • one example of SW-landscape

2.2. Understanding Similarity-Weighted (SW) through heat maps

Original-PIPE-landscape  /  norm_factor                =   SW-PIPE-landscape

6. Enrichment Calculation

  • Using the exact similar approach as given in original PIPE-sites paper by calculating the similarities between 2 protein pairs i.e. for every protein pair if DM = 1 from the lab site then we are predicting worst interaction site but if the DM = 0 when comparing to lab site then that would be an ideal prediction. 
    • Hence, we are using distribution of DM values representing enrichment as shown in these plots. 
  • We also used Dice-coefficient and Jaccard index (to compare with DM) but we found those values to be highly skewed because of the nature of our problem and therefore concluded those to be working well for this type of problem.

7. Insights for Walk Vs Panoramic view and Walk Vs Bright-Spot Detection

  • Walk algo works really good when there is a steep contrast in landscape-values (image)
  • Sim-Weighted normalizes the landscape-values (image) by a factor completely depending on the position/biding-site. This is not uniform for the entire image, meaning norm-factor is not a constant value for the image but for each pixel i,j its value is simProt(i)*simProt(j) This reduces the contrast of image overall reducing the accuracy/TruePositiveRate of Walk-Algo
  • Since Panorama is very similar to Walk-algo. It has the same limitation (of not working well after applying SW)
  • The intuition behind using Panaroma algorithm/unwrapping algorithm/walk algorithm 2.0 etc. is when finding out the domain of protein A we take an optimal range of domain of protein B but for the original walk algorithm to find out the domain of protein A we consider only one value into account for protein B.
